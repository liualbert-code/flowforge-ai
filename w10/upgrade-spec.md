\## spec.md 五區改造計畫

&#x20;

\### A 精神

要改嗎？(要)

\- 改成什麼？



v2 最大的改變，是從「教人用 AI 工具」變成「幫人建立 AI-native 的能力與工作方式」。

v1 比較像是在解決「怎麼用 AI 做事更快」，但 v2 更在意的是：「這個人有沒有真的變強？」

公司的核心開始從工具、產出、代工，慢慢轉向 workflow、problem solving 與 human × AI collaboration。



\- 為什麼？



我覺得以前比較像 AI 顧問／工具公司，但現在更像「能力放大公司」。

不是幫客戶把事情做完而已，而是幫他建立一套未來能持續進化的工作與思考系統。 



\### B 服務目錄

要改嗎？(要)

\- 改成什麼？



v2 的服務，最大的差別是「不再只是交付結果」，而是開始幫客戶建立可以長期運作的系統。

v1 比較像一次性的服務，例如幫客戶整理、生成、完成某件事情；但 v2 開始變成會陪客戶一起看 workflow 哪裡卡住、問題怎麼拆、AI 怎麼真正融進日常工作裡。



服務內容也開始加入：

\- workflow 診斷

\- AI-native 工作流設計

\- agent 協作系統

\- 長期陪跑

\- problem solving coaching



\- 為什麼？



我覺得 v2 最重要的改變，是「協作感」變強了。

不再只是客戶丟需求、公司交文件，而是一起建立 human × AI collaboration 的方式。

最後留下的不是一份報告，而是一套客戶之後還能持續使用的能力與系統。

&#x20;

\### C specialists

要改嗎？(要)

\- 改成什麼？



v2 的 specialists 最大的差別，是開始從「功能型工具」變成「真的像團隊成員」。

v1 的 specialists 比較像是：使用者下指令，agent 完成某個任務；但 v2 開始會記憶上下文、理解客戶狀態、知道目前卡在哪，甚至會主動發現問題。

而且 specialists 之間也開始具有協作能力，不再是各做各的，而是會互相辯論、互相補位、一起拆 problem，慢慢變成真正的 AI-native team。

v2 的 specialists 也不只是等待命令，而比較像「會一起思考的人」。



例如會：

\- challenge 使用者 workflow

\- 提醒盲點

\- 協助重新定義問題

\- 幫助建立 problem solving 方法



\- 為什麼？



我覺得 AI-native specialist 不應該只是執行工具，而應該成為長期協作夥伴。

真正有價值的不是「做完任務」，而是幫助客戶建立更好的思考與工作方式。

&#x20;

\### D COO 工作流

要改嗎？(要)

\- 改了什麼？



v2 的 COO workflow，最大的改變是：不再只是「收到需求 → 分派任務 → 交付結果」這種傳統流程。

現在比較像是先理解問題本身。

COO 不只是接案，而是會先診斷客戶真正卡住的地方，拆解 problem、分析 workflow，甚至判斷這是不是表面需求。

workflow 也開始變成動態協作系統。

不是固定 specialist 做固定事情，而是會根據問題類型去分配 agent、讓 specialists 互相協作、辯論與補位。

另外，v2 很強調「長期記憶」與「持續迭代」。

因為客戶不是一次性需求，而是持續進化的人，所以 workflow 也需要跟著更新。



\- 為什麼？



我覺得 v2 的 COO 已經不只是專案管理，而是在管理「客戶能力成長系統」。

真正重要的不是把任務做完，而是讓客戶的 workflow 與 problem solving 能力持續進化。

&#x20;

\### E deliverable 結構

要改嗎？(要)

\- 改成什麼？



v2 deliverable 最大的差別，是開始變成「可以持續運作的系統」，而不只是一次性的文件。

v1 比較像交報告、交簡報、交整理好的內容，事情完成後服務就結束了；但 v2 更在意的是：「客戶之後能不能自己持續使用這套系統？」

因此 deliverable 不再只是文件，而會開始包含：

\- workflow map

\- problem solving framework

\- agent collaboration structure

\- 長期 memory

\- 下一步行動系統



\- 為什麼？



我覺得 v2 很重要的是「可延續性」。

以前 deliverable 比較像成果物，現在比較像「能力基礎建設」。

客戶之後遇到新的問題時，還能繼續用這套系統去思考、協作與進化。

\## specialist 檔案改造計畫

&#x20;

\### 要重寫的 specialist

\- 檔名：workflow-architect.md

\- 為什麼要重寫：



v1 的 workflow architect 比較像工具與流程整理者，

但 v2 開始更重視 problem solving、workflow 診斷與 human × AI collaboration。

因此需要讓這個 specialist 從「流程整理工具」轉變成真正能理解客戶工作方式、協助建立 AI-native workflow 的角色。



\- 檔名：problem-solving-coach.md

\- 為什麼要重寫：



v1 比較偏向直接給答案或完成任務，

但 v2 更希望這個 specialist 能協助客戶拆解問題、重新定義問題，並建立長期 problem solving 能力。



因此它不再只是任務執行者，而是陪客戶一起思考與成長的 AI collaborator。 



\### 要砍的 specialist

目前沒有需要完全刪除的 specialist。



但 v1 偏向「功能型工具 agent」的設計，會在 v2 被重新轉化成 AI-native collaboration specialist。

&#x20;

\### 要新增的 specialist

\- 新檔名：ai-native-companion.md



\- 核心職責：



這個 specialist 的核心工作，是理解與追蹤客戶的成長與 workflow 演化。

它不只是記錄資料，而是長期觀察客戶怎麼做事、哪裡容易卡住、哪些 workflow 真的有效、哪些只是表面效率。

它會透過 long-term memory 跨專案理解客戶，記住以前的問題、使用習慣與能力變化，並在客戶再次出現相似卡點時主動提醒。

例如：「你現在又回到以前那個 workflow 混亂的狀態了，我們要不要先重新整理問題？」

它不是被動紀錄者，而是陪跑型 AI collaborator，會持續幫使用者優化 problem solving 與 human × AI collaboration 的方式。



\- 為什麼這個 specialist「只有 AI 在場才存在」：



因為人類顧問很難長期、即時、跨專案記住客戶所有互動脈絡，也很難在每一次任務中都把過去的卡點、偏好、工作方式與能力變化一起納入判斷。

但 AI 可以把每次互動變成共同記憶，持續追蹤客戶的 workflow 演化，並在不同任務之間建立連續性。

這種「長期記憶 + 持續陪跑 + 動態迭代 workflow」的角色，不是傳統公司裡原本就存在的職位，而是 AI 在場後才真正成立的原生型 specialist。

\## workflow 與 COO 改造計畫

&#x20;

\### 要新增的 slash command

\- workflow 名稱：/diagnose-workflow



\- 做什麼：



這個 workflow 最主要的目的，是「先看懂問題，再決定怎麼解問題」。

很多客戶提出的需求，其實只是表面症狀，真正卡住的可能是：

\- workflow 混亂

\- 資訊斷裂

\- AI 使用方式錯誤

\- problem solving 能力不足

\- 不知道真正問題核心是什麼



因此 workflow 一開始會先診斷：



\- 這是真問題還是假問題？

\- 卡點是在知識、執行、協作，還是 problem solving？

\- 哪些部分適合交給 AI？

\- 哪些部分需要 human decision？



之後才會動態分配 specialist。



它不是固定流程，而是會根據問題類型建立不同的 AI collaboration flow，讓 specialists 互相協作、補位與辯論。



這個 workflow 的最終目標，不只是完成任務，而是幫客戶建立更好的工作方式與思考方式。 



\### COO 工作流要不要改

\- 要

\- 改了什麼邏輯：



v1 的 COO 工作流比較像傳統專案流程：

「收到需求 → 分派任務 → 交付結果」。



但 v2 開始變成：

「診斷問題 → 分析 workflow → 動態分配 specialist → 持續迭代 → 建立長期 memory」。

COO 不再只是任務分派者，而更像 AI-native orchestration system。

它會根據客戶狀態、問題類型、workflow 與長期記憶去動態調整整個 collaboration flow。

因此 v2 的 COO 更像是在管理「人的能力成長系統」，而不只是管理任務。
